# 1. Introduction to System Design

## What is System Design?

System Design is the process of defining the **architecture, components, modules, interfaces, and data** for a system to satisfy specified requirements. It serves as a bridge between system requirements and implementation.

## Why is System Design Important?

| Aspect | Importance |
|--------|------------|
| **Scalability** | Handle growth in users and data |
| **Reliability** | System works correctly under various conditions |
| **Availability** | System is accessible when needed |
| **Maintainability** | Easy to update and fix |
| **Performance** | Fast response times and high throughput |
| **Cost Efficiency** | Optimal use of resources |

## Types of Requirements

### Functional Requirements
Specific behaviors or functions the system must perform:
- User authentication
- Data processing
- Report generation
- Payment processing

### Non-Functional Requirements
Quality attributes and constraints:
- Performance (response time < 200ms)
- Scalability (handle 1M users)
- Security (encryption, authentication)
- Availability (99.9% uptime)

|-----------|-------------|----------|
| **Round Robin** | Requests distributed sequentially | Equal server capacity |
| **Weighted Round Robin** | Based on server capacity weights | Different server specs |
| **Least Connections** | Routes to server with fewest connections | Long-lived connections |

# 15. Design Patterns

## Creational Patterns

| Pattern | Description | Use Case |
|---------|-------------|----------|
| **Singleton** | Single instance only | Configuration, Logging |
| **Factory** | Create without specifying class | Object creation abstraction |
| **Builder** | Step-by-step construction | Complex objects |
| **Prototype** | Clone existing objects | Object copying |

## Structural Patterns

| Pattern | Description | Use Case |
|---------|-------------|----------|
| **Adapter** | Makes incompatible interfaces work | Legacy integration |
| **Decorator** | Add behavior dynamically | Extending functionality |
| **Facade** | Simplified interface | Complex subsystems |
| **Proxy** | Placeholder for another object | Access control, caching |

## Behavioral Patterns

| Pattern | Description | Use Case |
|---------|-------------|----------|
| **Observer** | Notify dependents of changes | Event systems |
| **Strategy** | Interchangeable algorithms | Payment methods |
| **Command** | Encapsulate request as object | Undo/Redo |
| **State** | Behavior changes with state | Order status |

## System Design Checklist

| Component | Questions to Ask |
|-----------|-----------------|
| **Requirements** | Functional? Non-functional? Scale? |
| **Data** | Size? Growth rate? Access patterns? |
| **Storage** | SQL or NoSQL? Sharding? Replication? |
| **Caching** | What to cache? Strategy? Eviction? |
| **Load Balancing** | Algorithm? Layer 4 or 7? |
| **APIs** | REST? gRPC? GraphQL? Rate limiting? |
| **Security** | Authentication? Encryption? Authorization? |
| **Monitoring** | Metrics? Logging? Alerting? |
| **Failure** | What can fail? How to recover? |

